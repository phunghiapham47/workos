# WORKOS — Database Schema

## Overview

WorkOS uses Supabase for lightweight operational persistence.

Core principles:

- Minimal fields
- Fast interaction
- Low friction
- Personal operational tracking
- No enterprise complexity

Main tables:

- `projects`
- `tasks`

No current tables for:

- Auth profiles
- Teams
- Permissions
- Collaboration
- Activity logs
- Notifications

## Current Persistence Architecture

`WorkOSProvider` is the single source of truth for the UI.

On app start:

- Load `projects` from Supabase.
- Load `tasks` from Supabase.
- Map database rows into app models.

During actions:

- UI updates optimistically.
- Supabase mutation runs in the background.
- On mutation error, the affected state is rolled back and an error is logged.

Dashboard, Projects, and Tasks all read from the same provider state.

## Table: projects

Purpose:

Track operational projects, workflow status, revenue, paid amount, and collection exposure.

Current Supabase columns observed by the app:

| Field | Type | Required | Notes |
|---|---|---|---|
| id | uuid | yes | Primary key generated by app |
| created_at | timestamptz | yes | Auto generated by Supabase |
| name | text | yes | Project name |
| client | text | yes | Client name |
| status | text | yes | Workflow status |
| revenue | numeric | no | Revenue excl.VAT by default |
| paid | numeric | no | Total paid amount |
| outstanding | numeric | no | Existing DB column; app calculates outstanding itself |
| note | text | no | Short operational note |

Important:

- The deployed schema cache did not expose `updated_at` during verification.
- The app therefore does not write `updated_at`.

## Project Status Values

Current UI status values:

- Planning
- In Progress
- Review
- Payment
- Done

Legacy status values are normalized on load:

| Legacy value | Current UI value |
|---|---|
| Lead/Brief | Planning |
| In progress | In Progress |
| Nghiệm thu | Review |
| Chờ thanh toán | Payment |
| Pending Payment | Payment |

Done state:

- Lower visual priority.
- Treated as archived.
- Excluded from active pipeline count.

## Project Finance Logic

Project outstanding:

```text
max(revenue - paid, 0)
```

Project module metrics:

- Active Pipeline: projects excluding Done.
- Pipeline Value: revenue from active projects.
- Outstanding: outstanding amount from `Payment` projects only.
- Paid Received: sum of all paid amounts.

Dashboard:

- Reads from shared provider state.
- Uses project counts/statuses for operational scan.
- Does not use weighted progress.

## Table: tasks

Purpose:

Track lightweight operational tasks and reminders.

Tasks remain independent from projects.

Current Supabase columns observed by the app:

| Field | Type | Required | Notes |
|---|---|---|---|
| id | uuid | yes | Primary key generated by app |
| created_at | timestamptz | yes | Auto generated by Supabase |
| title | text | yes | Task content |
| due_date | date | no | Used for queue grouping |
| completed | boolean | yes | Source for UI done state |

Important:

- The deployed schema uses `completed: boolean`.
- The app does not write `tasks.status`.
- UI status is derived from `completed`.

## Task Status Mapping

| Database | UI |
|---|---|
| completed = false | To-do |
| completed = true | Done |

Task tabs derive from UI state and due date:

- Overdue
- Today
- Upcoming
- Done

## Task Input Structure

Input format:

```text
DD/MM - Task content
```

Examples:

- `29/05 - Follow payment Coca`
- `30/05 - Send proposal Nutifood`
- `05/10 - Check client payment`

If no date is provided:

- Assign current day.

## Relationships

No direct relationships in v1.

Tasks remain independent from projects to preserve operational speed and avoid premature complexity.

## Schema Constraints Going Forward

Preserve current behavior unless a migration is intentionally planned:

- Do not add auth tables yet.
- Do not add team/collaboration tables.
- Do not change `tasks.completed` without migrating provider mapping.
- Do not rely on `projects.updated_at` until the deployed schema exposes it.
- Keep frontend calculations lightweight and explicit.

## Verification Expectations

Before deployment when persistence is touched:

1. Add project, refresh, project remains.
2. Edit project, refresh, edit remains.
3. Delete project, refresh, deleted project stays deleted.
4. Add task, refresh, task remains.
5. Complete task, refresh, completed state remains.
6. Undo task, refresh, restored state remains.
7. Delete task, refresh, deleted task stays deleted.
